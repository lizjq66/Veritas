# VaR Formula Audit — 2026-04-23

**Status:** draft; no code change proposed yet. Written to clarify the
real scope of `Veritas/Gates/PortfolioGate.lean`'s `portfolioVarBound`
before we commit to further VaR work.

**Scope:** the formula defined by `portfolioVarBound`, the claim made
by its current docstring, and the consequences for `slice 8`'s
`certificate_approve_final_within_gate3_var_bound`.

**Bottom line (one sentence):** the formula **is** a valid
triangle-inequality upper bound — but on a narrower quantity than the
docstring claims, and it is **not** an upper bound on the full
portfolio's return standard deviation `√xᵀΣx` in general.

---

## 1. What `portfolioVarBound` actually computes

Current definition (simplified):

```
portfolioVarBound port p
  = Σᵢ |xᵢ|·σᵢ·|ρ₀ᵢ|  +  |x₀|·σ₀
```

where, for each existing position `i`:

- `xᵢ = entryPrice_i · size_i` (signed notional; the formula takes `|xᵢ|`)
- `σᵢ = volatility_i` (assumed ≥ 0 by slice 8)
- `ρ₀ᵢ = correlationBetween(port.correlations, pos.asset, p.asset)` —
  absolute value, clipped to `[0, 1]`; same-asset defaults to `1`,
  unknown cross-asset defaults to `0`

and for the proposal:

- `x₀ = p.notionalUsd`
- `σ₀ = p.volatility`

## 2. What the docstring currently says

> "This is an *upper bound* on the quadratic-form VaR `√xᵀΣx` via the
> triangle inequality"

This reads as: "for any covariance structure `Σ` on existing positions
+ proposal, the formula ≥ `√xᵀΣx`."

That reading is **wrong in general**. The formula doesn't use any
correlation information between *existing* positions (the only
correlations consulted are `ρ₀ᵢ` — between each existing and the
proposal).

## 3. What the formula **is** a valid upper bound on

Define the **proposal-axis projected exposure**:

```
E_proj = x₀·σ₀ + Σᵢ xᵢ·σᵢ·ρ₀ᵢ        (signed)
```

Then by the standard triangle inequality:

```
|E_proj| ≤ |x₀|·σ₀ + Σᵢ |xᵢ|·σᵢ·|ρ₀ᵢ|  =  portfolioVarBound
```

So **`portfolioVarBound ≥ |E_proj|` unconditionally.** This is the
theorem that would actually be provable in Lean, and it IS a useful
risk concept: "total absolute exposure projected onto the new asset's
return factor."

What it is NOT: a bound on the portfolio's full 1-day return
standard deviation `√xᵀΣx`, except in the very narrow cases in §4.

## 4. Scenarios where `portfolioVarBound ≥ √xᵀΣx` (correct)

### 4a. Single asset throughout

All existing positions and the proposal share one asset.
All `ρ`s are `1` by construction of `correlationBetween`.

```
portfolio return = (Σ xᵢ + x₀) · r_asset
√Var            = |Σ xᵢ + x₀| · σ_asset
formula         = (Σ |xᵢ| + |x₀|) · σ_asset   ≥ the above
```

### 4b. No existing positions

Formula reduces to `|x₀|·σ₀ = √Var` exactly.

### 4c. Proposal-uncorrelated single existing + proposal

Existing `x₁` in asset A, proposal `x₀` in asset B, `ρ₀₁ = 0`.

Formula = `|x₀|·σ₀`. True stddev = `√(x₁²σ₁² + x₀²σ₀²)`. Formula <
true stddev unless `x₁ = 0`. **NOT a correct bound.** Covered in §5.

### (That's it.)

Outside §4a and §4b, the formula is **coincidentally** a correct
upper bound on `√xᵀΣx` only if the existing positions happen to be
perfectly correlated with the proposal's asset (which subsumes them
into §4a), or if correlations conspire to make the projected
exposure equal the full stddev. There is no family of
*realistic* correlation matrices outside §4a–4b where the
inequality holds robustly.

## 5. Counter-examples where `portfolioVarBound < √xᵀΣx`

### CE-1: Two perfectly correlated existing, uncorrelated with proposal

- Existing: `x₁ = x₂ = 1`, `σ₁ = σ₂ = 1`, `ρ₁₂ = 1`
- Proposal: `x₀ = 0`, any `σ₀`
- `ρ₀₁ = ρ₀₂ = 0`

```
True Var = 1 + 1 + 2·1·1·1 = 4   →   √Var = 2
Formula  = 1·1·0 + 1·1·0 + 0    = 0
```

Formula underestimates by **factor ∞** — says "no risk", truth is
`√Var = 2`.

### CE-2: Single existing in an uncorrelated asset

- Existing: `x₁ = 1` in asset A, `σ₁ = 1`
- Proposal: `x₀ = 0` in asset B, `ρ₀₁ = 0`

```
True √Var = 1
Formula    = 0
```

Formula UNDERESTIMATES. The docstring claim fails on this trivial
two-asset case.

### CE-3: Moderate correlations

- `x₁ = x₂ = 1`, `σ₁ = σ₂ = 1`, `ρ₁₂ = 0.5`
- `ρ₀₁ = ρ₀₂ = 0.5`, `x₀ = 0`

```
True Var = 1 + 1 + 2·0.5 = 3     →   √Var ≈ 1.73
Formula  = 1·1·0.5 + 1·1·0.5 + 0 = 1
```

Formula underestimates (1 < 1.73).

## 6. Status of slice 8's theorem in each interpretation

The slice-8 theorem:

```
certificate_approve_final_within_gate3_var_bound :
  Approve + (0 < dailyVarLimit) + (0 ≤ p.volatility) →
    portfolioVarBound port { p with notionalUsd := finalNotional }
      ≤ dailyVarLimit
```

is **literally true** regardless of which interpretation of
`portfolioVarBound` we adopt. The theorem is a statement about the
value of the formula expression, not about real-world VaR.

Its **real-world meaning** depends on which interpretation the
operator/caller attaches to `dailyVarLimit`:

| Caller's interpretation of `dailyVarLimit`                              | Slice-8 guarantee they actually get                                                     |
| ----------------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| "Upper bound on proposal-axis projected exposure I'll tolerate"         | **Accurate.** Formula is exactly an upper bound on that quantity; theorem enforces it. |
| "Upper bound on full portfolio 1-day return stddev `√xᵀΣx`"             | **Misleading.** Formula is not an upper bound on that quantity; false sense of security (see §5). |

Under the **narrow (correct) interpretation**, slice 8 remains a
valid contribution. Under the **broad (docstring-advertised)
interpretation**, slice 8 is formally true but operationally
over-claimed.

## 7. Remediation options

### Option A — Fix the docstring; keep formula and theorem unchanged

- Change `portfolioVarBound`'s docstring in
  `Veritas/Gates/PortfolioGate.lean` to describe what it actually
  bounds (§3).
- Update `certificate_approve_final_within_gate3_var_bound`'s
  docstring and the corresponding theorem-registry entry to the same
  narrower interpretation.
- Add a short note to `docs/paper/veritas.tex` if the paper mentions
  quadratic-form VaR.
- **No code or proof changes.**

  | Pro                                         | Con                                                                 |
  | ------------------------------------------- | ------------------------------------------------------------------- |
  | 10-minute slice; nothing breaks             | Users must understand the narrower guarantee                        |
  | Slice-8 theorem stays intact                | Does not deliver the strong guarantee the original docstring hinted |
  | Preserves fast, Rat-arithmetic bound        | Some callers will want a stronger bound; leaves a follow-up open    |

### Option B — Replace formula with a conservative always-valid bound

Swap `portfolioVarBound` for the correlation-ignoring sum:

```
portfolioVarBound_safe port p
  = Σᵢ |xᵢ|·σᵢ  +  |x₀|·σ₀
```

This is an upper bound on `√xᵀΣx` unconditionally (worst-case ρ=1
across all pairs, triangle inequality). Slice-8 theorem needs a
signature tweak; proofs stay structurally similar (the correlation
weight drops out). Monotonicity helper still applies.

  | Pro                                               | Con                                                            |
  | ------------------------------------------------- | -------------------------------------------------------------- |
  | Delivers what the docstring originally claimed    | Very conservative; legitimately hedged portfolios may get rejected |
  | Still Rat-arithmetic, fast                        | Breaks any user setting `dailyVarLimit` to the previous semantics |
  | Correlation-weighted bound can return as a separate, optional check | Rewrite of slice 2 + slice 8 docstrings/commit messages / theorem registry entries |

### Option C — Keep current formula; add a true quadratic-VaR gate alongside

- `portfolioVarBound` stays (renamed to `projectedExposureBound`).
- New `portfolioQuadraticVar` that actually computes `xᵀΣx` (needs
  full pairwise correlation table on existing positions).
- New gate branch chooses which bound to enforce based on
  `c.dailyVarLimitSemantics = projected | quadratic`.
- New theorems for each branch.

  | Pro                                       | Con                                                               |
  | ----------------------------------------- | ----------------------------------------------------------------- |
  | Callers opt into whichever they want      | Much more Lean/Python/test work; multi-slice investment           |
  | Honest story for both risk cultures       | Needs square-root or iterative approximation for `√xᵀΣx`; breaks Rat-only arithmetic |
  | Positions us to claim strongest guarantee | Covariance matrix needs to be a caller input                      |

### Options summary

```
A (docstring only)   : ship today, honest about narrow semantics
B (stronger bound)   : multi-hour slice, breaks downstream meaning,
                       but delivers "√xᵀΣx" guarantee
C (both bounds)      : multi-slice v0.4 feature, maximum correctness
                       at maximum complexity
```

## 8. Recommended path

**Do Option A now.** Reasons:

1. The formula is not broken — it's a meaningful risk metric with a
   clean triangle-inequality proof. What's broken is the docstring
   overclaim.
2. Slice 8's theorem is real work and should not be thrown away;
   Option A preserves it intact, under an honest interpretation.
3. A conservative Option-B swap would reject otherwise-acceptable
   hedged portfolios; surprising callers with a stricter gate is
   itself a correctness issue.
4. If stronger guarantees become a product requirement, Option C is
   the right path and it can be planned as a v0.4 feature, not
   shoehorned into the v0.3 tail.

**Concrete Option-A deliverable** (when you greenlight):

- One commit: `docs: correct VaR formula interpretation` with
  - Docstring rewrites: `portfolioVarBound`,
    `checkPortfolio_approve_respects_var_bound`,
    `checkPortfolio_resize_respects_var_bound`,
    `certificate_approve_final_within_gate3_var_bound`
  - Theorem-registry entries: same two (the three most relevant
    theorems' `statement_natural_language` fields)
  - Short paragraph in README's caller-side section explaining
    the narrow semantics of `dailyVarLimit`
  - Reference to this audit memo
- No code changes. No theorem changes. No test changes.

If you prefer Option B or C, say so and I will draft the slice plan
separately.

## 9. Open questions for the operator

1. What interpretation of `dailyVarLimit` do you want to publish?
   "projected exposure" or "full-portfolio stddev"?
2. Is there an existing caller who has already pinned a value to
   `dailyVarLimit`? (If yes, changing semantics is a public-contract

3. Is `correlationBetween` meant to be absolute-valued (current) or
   signed? Signed correlations would let the projected-exposure bound
   distinguish hedging from doubling-down, at the cost of needing

   use `|entryPrice·size|`).
